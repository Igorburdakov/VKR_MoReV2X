# Эксперименты

Все эксперименты у меня используют одну и ту же карту (HIGHWAY-80 dense, 80 машин,
плотность 10.58 veh/lane-km) и trace на 100 секунд. На каждый параметрический набор
запускается 3 seed-реплики (runNo = 1, 2, 3).

Общие параметры для всех групп:
- Vehicles = 80
- MCS = 13
- ChannelBW = 20 МГц
- SavingPeriod = 1.0 с
- simTime = 100 с

CBR (как контролируем загрузку канала):
- **CBR ~ 50%**: период генерации CAM = 1.35 * RRI, HARQ = 1
- **CBR ~ 95%**: период генерации CAM = RRI, HARQ-передачи на один TB = 2 (нужно для загрузки ресурсов)

При смене нумерологии (0, 1, 2) длительность слота меняется (1.0, 0.5, 0.25 мс).
Чтобы нагрузка на канал оставалась сопоставимой между нумерологиями, для 1, 2
RRI и CAM пропорционально уменьшал. Размер subchannel при этом тоже менял
(100 RB, 50 RB, 20 RB для полосы пропускания BW=20 МГц). Так в каждом случае получал один подканал на слот.

## G1 - Numerology * pKeep * CBR

Влияние нумерологии и персистентности при двух уровнях нагрузки.

Параметры:
- mu = {0, 1, 2}
- pKeep = {0.0, 0.4, 0.8}
- CBR = {50%, 95%}
- txPower = 23 dBm

RRI и CAM по mu:

| mu | RRI | CAM (CBR50) | CAM (CBR95) | subch |
|----|-----|-------------|-------------|-------|
| 0  | 100 | 135         | 100         | 100   |
| 1  | 50  | 67          | 50          | 50    |
| 2  | 25  | 34          | 25          | 20    |

Итого: 3 * 3 * 2 * 3 = 54 прогона.

## G2 - txPower * pKeep * CBR (mu = 0)

Влияние мощности передатчика и персистентности.

Параметры:
- txPower = {17, 20, 23} dBm
- pKeep = {0.0, 0.4, 0.8}
- CBR = {50%, 95%}
- mu = 0, RRI = 100, subch = 100

Итого: 3 * 3 * 2 * 3 = 54 прогона.

## G3 - RRI * pKeep * CBR (mu = 0)

Влияние интервала резервирования и персистентности.

Параметры:
- RRI = {20, 50, 100} мс
- pKeep = {0.0, 0.4, 0.8}
- CBR = {50%, 95%}
- mu = 0, txPower = 23, subch = 100

CAM по RRI:

| RRI | CAM (CBR50) | CAM (CBR95) |
|-----|-------------|-------------|
| 100 | 135         | 100         |
| 50  | 67          | 50          |
| 20  | 27          | 20          |

Итого: 3 * 3 * 2 * 3 = 54 прогона.

---

## Итого

162 прогона. Загруженность по CBR корректировал по тестовым запускам с разным количеством UE (брал 60, 70, 80, 100) и при разной частоте генерации CAM сообщений. Пробовал брать CAM как меньше, так и больше диапазона RRI. Если CAM меньше RRI, то нужно снижать и сам RRI, иначе сообщения в основном накапливаются в MAC-буфере и не дают ожидаемого роста CBR.

## Где брать значения CAM

Все CAM-значения подобрал заранее на основе калибровочных тестов. Формула
приближения для других точек: CBR ~ 0.73 × (100 / CAM)^1.31 (при mu=0, RRI=100, 80 UE).
